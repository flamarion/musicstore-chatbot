Why ReAct

The job is "pick the right tool and answer." A prebuilt agent ⇄ tools loop fits exactly — no bespoke graph.

Why middleware

Limits, retry, PII, guardrails, consent are cross-cutting. Middleware keeps them out of tool logic.

Why a checkpointer

Production agents are multi-turn. LangGraph owns state; the client stays thin.

Named in the brief · "how does a customer see only their own info?"

Identity is email-only, enforced in code.

The two-"Luis" problem

Two customers are named Luis. List both and let the user pick → that's impersonation. So a name never unlocks data, and we never enumerate matching accounts.

Personal data releases only on a unique, exact email match — resolve_customer_for_pii(), in the tool layer, not the prompt.

Consent, once

The first time an email is used, HumanInTheLoop interrupts for approve/reject. Approve once → later lookups for that email don't re-prompt (the when predicate scans history).

Defense in depth

PIIMiddleware redacts emails the bot would echo in replies — but not input (apply_to_input=False).

The trap

Email is the identifier. Redact it on input and the tool sees [REDACTED] — no account ever matches, the whole gate breaks. So input off, output on.

Say it"A prompt can be talked around; a code gate can't. Identity lives in the tool layer — and consent is a real graph interrupt, not a prompt trick."
Reliability engineering · via middleware

Two custom guards. Five built-ins. Each a production concern.

custom

Profanity + Topic guards

Short-circuit off-topic / abusive input before the LLM — zero cost. Prompt "stay in scope" is the real backstop.

built-in

ModelCallLimit

run_limit=8 — a runaway tool loop can't burn cost or hang.

built-in

Summarization

Condenses history past a token budget. Long chats won't blow the window.

built-in

PII redaction

Emails masked in replies only — privacy without breaking lookups.

built-in

ModelRetry

Retry + backoff, friendly message on hard failure. Blips don't crash the chat.

built-in

HumanInTheLoop

The consent gate — a real interrupt/resume before personal-data tools.

  • See it. One real turn → the trace tree: every hop with inputs, outputs, latency, tokens — and honest run types, not one generic chain.
  • Debug it. Pull up a bad run, find the wrong hop — a tool arg, a guard branch — instead of guessing. This is the gap you fell into.
  • Trust it. Turn real traces into a dataset; run evals (LLM-judge, exact-match) to catch regressions before shipping.
  • Operate it. Latency, cost, error rate, feedback over time. Filter tool runs by tag:pii for an audit view.

Friction log · honesty reads as competence

The Lang* stack was new to me. Here's what bit.

Learning the whole stack

LangChain 1.x, LangGraph, middleware, LangSmith run types — all new. The docs-MCP + tracing shortened the loop. Exactly the customer's starting point.

PII input-masking broke lookups

PIIMiddleware redacts input by default — it erased the email before the tool ran. Fix: output-only.

Consent asked every turn

HITL re-prompted on each PII tool. Fix: the when predicate scans history → consent once per email per thread.

Middleware typing broke overloads

Heterogeneous middleware + an invariant StateT made create_agent fail type-checking. Fix: annotate the list explicitly.

Chat-UI's baked localhost URL

A NEXT_PUBLIC_ API URL is inlined at build time — "localhost:3000" broke on a remote host. Fix: run local / rebuild with the real host.

Deep Agents temptation

Evaluated create_deep_agent, rejected it as overkill for single-turn lookups. Knowing when not to reach for a tool is a decision too.
