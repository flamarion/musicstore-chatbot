Named in the brief · "a customer sees only their own info"

Identity is email-only, enforced in code.

The two-"Luis" problem

Two customers are named Luis. List both and let the user pick → impersonation. So a name never unlocks data, and we never enumerate accounts — release only on a unique, exact email match, in the tool layer, not the prompt.

Consent, once

First use of an email → HumanInTheLoop interrupts for approve/reject. Approve once → later lookups for that email don't re-prompt.

Defense in depth

PIIMiddleware redacts emails the bot would echo in replies — but not input.

The trap

Email is the identifier. Redact it on input and the tool sees [REDACTED] — no account matches, the gate breaks. So input off, output on.

Reliability engineering · via middleware

Two guardrails. Five built-ins. Each a production concern.

Profanity + Topic guards custom ×2
Short-circuit abusive / off-topic input before the LLM — zero cost.
ModelCallLimit built-in
A runaway tool loop can't burn cost or hang.
Summarization built-in
Condenses history past a token budget — long chats stay in-window.
PII redaction built-in
Emails masked in replies only — privacy without breaking lookups.
ModelRetry built-in
Retry + backoff, friendly message on failure — blips don't crash.
HumanInTheLoop built-in
A real interrupt/resume consent gate before personal-data tools.

PII input-masking broke lookups
It erased the email before the tool ran. Fix: output-only.
Consent asked every turn
Fix: the when predicate scans history → once per email / thread.
Middleware typing broke overloads
Invariant state type failed create_agent. Fix: annotate the list.
Chat-UI's baked localhost URL
A build-time env broke on a remote host. Fix: run local / rebuild.
Deep Agents temptation
